# Server 1.0.0 / Android and Wear 0.1.0 release candidate

Status: locally validated, not released. This is the working release note for the first private
self-hosted native dogfood build. Test-signed phone/Wear artifacts and a clean encrypted restore
drill are complete. Permanent signing, Play declarations, physical Galaxy Watch Ultra evidence,
and the four-week exclusive dogfood period remain release gates.

## Highlights

- Self-hosted web/backend stack with Caddy or Traefik and external or in-stack Postgres.
- Manual/provider/barcode food logging, My Foods, recipes, weight trends, goals, and Lose It import.
- Native Android food, weight, goals, account, offline outbox, notifications, and self-host selection.
- Optional read-only Health Connect activity synchronization with observation-only calorie context.
- Wear OS pairing, cached daily summary, quick food, weight, completion, undo, offline outbox, Tile,
  reminders, rotary input, and continue-on-phone handoff.
- Versioned account export and password-confirmed deletion on web and Android, including a public hosted-service
  request page, Health Connect cleanup, and correlated paired-watch erase acknowledgement with durable recovery
  guidance when device cleanup cannot be confirmed.
- Encrypted Postgres backups, guarded restore tooling, readiness checks, structured diagnostics, and
  canonical release compatibility metadata.

## Database upgrade notes

Back up Postgres and verify the encrypted artifact before upgrading. Application startup runs
`prisma migrate deploy` before it becomes ready; a failed migration prevents the HTTP process from
opening and is retried only within the configured startup limit.

The native foundation and activity/watch releases add migrations 0021 through 0028. Migration 0029
changes browser push ownership from account-only to globally unique endpoint plus browser session.
Existing browser push rows are deliberately deleted because their owning session cannot be inferred
safely. Users who want web push must turn browser reminders off and on once after the upgrade. In-app
reminders, SSE refresh, Android native push, and watch-local reminders are unaffected.

## Compatibility

- Server API: v1; the legacy `/api` alias remains available during native migration.
- Minimum phone: 0.1.0.
- Minimum Wear: 0.1.0.
- Phone and Wear must share the same application ID and signing certificate for Data Layer pairing.
- One application process is supported per self-hosted instance; horizontal replicas are not.

## Rollback

Stop the application before rollback. Reverting only the application image is acceptable only when
the target version explicitly supports the already-applied schema. Do not assume an older image can
write new required columns or understand newer synchronization contracts.

An exact rollback across migration 0029 requires restoring the pre-upgrade database backup. Rolling
back only the image does not recreate revoked browser push rows. After any rollback, verify readiness,
authentication, one food write, one weight write, export, native refresh, and phone/watch version
compatibility before reopening the instance.

## Known release-candidate limitations

- Android and Wear native UI is English-only; the web UI retains en/es/fr/ru support.
- Activity does not automatically change the calorie target.
- Native remote push is disabled unless the operator explicitly enables Expo for private/internal use.
- Direct FCM, horizontal scaling, automatic activity-target adjustment, and macro goals are not in this release.
- Clean local phone and Wear Gradle artifacts, final merged permissions, explicit backup/cleartext
  state, Android 16 phone emulator smoke, Wear OS 6 emulator smoke, and a representative encrypted
  restore drill are complete. Permanent/Play signing, physical Galaxy Watch Ultra validation, a
  true upgrade from a distributed predecessor, and long-duration offline/dogfood evidence remain.
